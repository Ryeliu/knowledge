# 会议纪要整理 Skill

## 输入

音频文件路径：`$ARGUMENTS`

## 执行流程

### Step 1：转写

调用 MCP 工具 `transcribe_audio`，传入音频文件路径，获取带说话人标签的转写文本。

如果用户直接提供了转写文稿（而非音频文件），跳过此步直接进入 Step 2。

### Step 2：预处理与理解

**通读全文，建立上下文地图（先不输出）：**

- **人物关系网**：谁是主导者（话最多的人）、谁是特邀嘉宾、谁是执行者
- **组织架构**：涉及哪些机构、部门、公司，彼此什么关系
- **话题锚点**：每次话题转换的关键词/触发句
- **情绪信号**：兴奋点、分歧点、决策点

**处理转写噪声：**

| 噪声类型 | 处理方式 |
|----------|---------|
| 错别字/同音字 | 根据上下文修正 |
| 口语化表达 | 提取核心语义，书面化 |
| 重复/回声 | 仅保留首次完整表述 |
| 说话人标识错误 | 根据内容语境推断真实说话人 |
| 无意义语气词/打断 | 直接过滤 |

**说话人识别策略（当只有"说话人01/02"时）：**
1. 通过自称、他称推断身份
2. 通过专业领域推断角色
3. 通过其他人的称呼
4. 无法确定则保留原标识，不强行猜测

### Step 3：话题线程提取

识别主话题线程，线下讨论通常呈**树状结构**：
- 标记每段对话属于哪个话题线程
- 同一话题在不同时间段反复出现 → 合并整理
- 识别话题之间的因果/依赖关系

**区分内容层级：**

| 层级 | 特征 | 处理 |
|------|------|------|
| 战略级 | 方向性判断、长期愿景 | 放最前面，概括性表述 |
| 项目级 | 具体要推进的事项 | 详细展开，明确责任人和时间 |
| 执行级 | 操作细节、资源对接 | 归入对应项目下 |
| 闲聊级 | 跑题、插科打诨 | 过滤，除非包含有价值观点 |

### Step 4：输出会议纪要

按以下结构输出，保存到 `meetings/YYYY-MM-DD-标题.md`：

```markdown
# [会议标题] 会议纪要

- **日期**：
- **参与人**：[姓名列表]
- **关联项目**：[项目列表]

## 一、总体概览
用 1-3 段话概括核心框架，让没参会的人 30 秒理解全局。

## 二、重点议题（按优先级排列）

### 议题1：[名称]
- **目标**：
- **方案要点**：
- **合作方/资源**：
- **关键讨论**：（保留有价值的观点碰撞）
- **待解决问题**：

### 议题2：[名称]
...

## 三、资源与合作方梳理
（表格列出外部资源、合作方、关键人物及角色）

## 四、精彩观点集锦
提取 3-5 条最有洞察力的发言，用引用格式，标注说话人。
选择标准：有创新性、有启发性、代表重要判断。

## 五、下一步行动
- [ ] 谁 负责 什么（截止日期）
- [ ] ...

---

> **核心一句话**：[一句话提炼本次会议最核心的结论]
```

### Step 5：质量检查

输出前逐项检查：

**完整性**：所有主要话题覆盖？所有项目记录？关键人物列出？行动项无遗漏？

**准确性**：口语/方言理解正确？数字金额正确？人名公司名正确？观点归因正确？

**可操作性**：读者能否不看原文理解要点？行动项具体到可执行？优先级明确？

### Step 6：录入知识库

1. 保存会议记录到 `meetings/YYYY-MM-DD-标题.md`
2. 将会议中涉及的人物、项目信息更新到对应公司的 `companies/公司名/README.md`
3. 如果涉及新公司，创建 `companies/公司名/` 目录和 README.md
4. 调用 MCP 工具 `index_knowledge()` 更新向量索引

### Step 7：处理完成后

将音频文件移动到 `inbox/processed/`，询问用户是否删除。

## 常见陷阱

- **伪共识**：有人说"对对对"不代表决策，只有主导者明确拍板才记为决策
- **多次提及 ≠ 高优先级**：基于"是否有具体行动计划"判断优先级
- **客套话 vs 真承诺**：只有配套了时间和方式的承诺才记入行动项
- **术语/代称**：首次出现时括号内给出全称或释义
- **幽灵语句**：与前后文完全不连贯的孤立句子直接忽略

## 超长录音处理

如果转写文稿超长：
1. 分段阅读，每段提取"话题关键词 + 关键信息点"
2. 合并所有段的提取结果，按话题聚合
3. 基于聚合结果生成最终纪要
